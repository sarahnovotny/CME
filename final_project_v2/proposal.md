# Project Proposal: Critical-but-Fragile — Mapping Fragility Across Functional Categories of Open Source Infrastructure

> The original proposal title was *"Where Is the Market Failure Worst?"*. It was retitled to align with the final report and to avoid the implicit causal framing — this analysis is descriptive, not a market-failure identification strategy.

**Research question.** The European software industry depends on open source packages maintained by small, often unpaid teams — a well-documented public goods under-provision problem. But fragility is not uniform across the software supply chain: some functional categories of infrastructure (e.g., cryptography, web frameworks) attract corporate investment and foundation funding, while others (e.g., compression utilities, build tooling) remain chronically under-resourced. Which functional categories of load-bearing open source infrastructure are most fragile, how much EU software-sector economic value sits above them, and where should public maintenance investment be prioritised?

**Data sources.** (1) Libraries.io v1.6.0 (Zenodo), a publicly available dependency graph dataset covering 4.6 million packages across seven ecosystems (npm, Maven, PyPI, NuGet, RubyGems, Packagist, Cargo), providing dependent counts, contributor counts, release timestamps, issue metadata, and package description text; (2) Eurostat gross value added by NACE Rev.2 sector (nama_10_a64), retrieved programmatically via the DBnomics Python API, to quantify the total EU software-sector GVA exposed to this infrastructure layer.

**Tentative methods.** We construct composite criticality and fragility scores for each load-bearing package (>=100 downstream dependents) using weighted additive combinations of log-normalised dependency counts, contributor concentration, release staleness, and issue backlog severity. To characterise *what* the risk universe does, we apply text-as-data methods to package descriptions: TF-IDF vectorisation followed by topic modelling (Latent Dirichlet Allocation or Non-negative Matrix Factorisation) to discover functional clusters (e.g., cryptography, networking, data serialisation, testing, build tooling). We then analyse how fragility and criticality vary across these clusters, identifying where in the software supply chain the public goods under-provision is most severe. Total EU software-sector GVA from Eurostat provides the economic denominator — the value at risk above the fragile infrastructure layer — and the maintenance funding gap is estimated per functional cluster. To quantify estimation uncertainty, we use Monte Carlo bootstrap simulation (1,000 replicates), resampling packages with replacement and recomputing cluster-level fragility distributions and funding gaps. Results are presented as a criticality-vs-fragility scatter plot coloured by topic cluster, a cluster-level fragility comparison chart, bootstrap confidence intervals, and summary tables.

## How the Plan Changed During Execution

Several aspects of the project evolved as we encountered the data and ran the analysis.

**Topic modelling: NMF over LDA, and k=25 over k=10.** We proposed "LDA or NMF" as alternatives. In practice, NMF produced substantially more interpretable topics on these short technical descriptions (often a single sentence), so we adopted it exclusively. We initially ran with 10 topics but found that a single "core/package/API" catch-all cluster absorbed 21% of all packages and 40% of the risk universe, rendering the cluster-level analysis uninformative. A reconstruction error sweep (k=5..25) showed steady but diminishing error reduction with no sharp elbow, so we selected k=25 based on manual inspection of topic quality: at k=25 no cluster exceeds 7.5% of the corpus and the topics resolve into distinct functional categories.

**The topics are functional infrastructure categories, not industry sectors — and one is a residual.** The proposal anticipated clusters like "cryptography, networking, data serialisation." What NMF actually discovered are functional categories of *open source infrastructure*: build tooling (bundlers), JS transpilation (Babel), CSS & styling, cloud & API clients, code linting, test runners, HTTP & networking, and so on. The proposed industry framing was replaced with this functional framing in the report. Two of the 25 clusters (Topic 0 and Topic 24) are residuals that absorb packages with uninformative descriptions; Topic 0 in particular is dominated by legacy Java/Maven infrastructure rather than the TypeScript-ecosystem packages its top NMF keywords might suggest. We assigned human-readable labels to all 25 topics after inspecting their actual package composition rather than relying on the raw NMF keyword lists, and we report Topic 0 explicitly as "Generic/legacy infrastructure (residual)" — see `PROJECT_OVERVIEW.md` §11 for the full composition analysis.

**Criticality scoring without OpenSSF.** The proposal did not mention the OpenSSF Criticality Score, but an earlier iteration of the project incorporated it (60/30/10 weighting with Libraries.io dependency counts). For the final submission we dropped the OpenSSF component to keep the pipeline self-contained — it requires a separate 154MB download from Google Cloud Storage that is not part of the Libraries.io dataset. Criticality is therefore driven entirely by Libraries.io dependent counts (65/35 package/repository weighting), which means npm coverage is more complete than enterprise ecosystems like Maven and NuGet. This is acknowledged as a limitation.

**GVA as context, not a calibrated VaR.** An earlier version of the project calibrated the exposure index against the log4shell remediation cost to produce a euro-denominated "value at risk." This required a circular calibration anchor (the total VaR was an input, not an output) and was overly complex for the scope of a course project. In the final version, Eurostat GVA serves as economic context — how large is the software sector that depends on this infrastructure? — rather than as a denominator in a calibrated risk metric. The funding gap (EUR 22.2M) is presented in absolute terms and per cluster, not as a ratio.

**Data cleaning: the CSV off-by-one.** The Libraries.io CSV has 59 header columns but 60 data columns due to a trailing comma. This caused all columns to be misaligned when read with pandas' default parser. The fix — reading the header with Python's csv module and appending a dummy column — was not anticipated in the proposal but was essential to producing correct results. Missing contributor data (28% of packages) is imputed as 1 (single maintainer), a conservative assumption that treats missing data as evidence of fragility.
